
# 概述
消息队列 高吞吐 分区 复制 容错  
可靠性 - Kafka是分布式，分区，复制和容错的。

可扩展性 - Kafka消息传递系统轻松缩放，无需停机。

耐用性 - Kafka使用分布式提交日志，这意味着消息会尽可能快地保留在磁盘上，因此它是持久的。

性能 - Kafka对于发布和订阅消息都具有高吞吐量。 即使存储了许多TB的消息，它也保持稳定的性能。

producer consumer 生产者消费者 发送和接收方  
broker 实例 主机 服务  
topic 主题 消息的分类 消息队列的名字  
partition 分区 topic进一步细分 实际是append log文件  可以备份 存放在各个broker上 不支持随机读写  
offset 偏移量 消息在分区上的位置 long类型数字 标记唯一一条消息  
replicated 副本 选举一个server做leader负责对该分区的读写 其他作为follower只做同步。有多少patition就有多少leader，使用 ZK 在 Broker 中选出一个 Controller，用于 Partition 分配和 Leader 选举  

kafka 是一个基于发布-订阅的分布式消息系统（消息队列）  
Kafka 面向大数据，消息保存在主题中，而每个 topic 有分为多个分区  
kafka 的消息数据保存在磁盘，每个 partition 对应磁盘上的一个文件，消息写入就是简单的文件追加，文件可以在集群内复制备份以防丢失
即使消息被消费，kafka 也不会立即删除该消息，可以通过配置使得过一段时间后自动删除以释放磁盘空间  
kafka依赖分布式协调服务Zookeeper，适合离线/在线信息的消费，与 storm 和 spark 等实时流式数据分析常常结合使用，2.8.0之后逐渐舍弃zookeeper，使用KRaft模式管理。  
Raft 算法是一种用于分布式系统的共识算法，它能够确保集群中的各个节点对数据的一致性达成共识。当有新的 Broker 加入或原有的 Broker 出现故障时，集群会通过 Raft 算法自动进行选举和协调，确保数据的可靠存储和传输。

# 模式
## 点对点  
一个或多个消费者可以消耗队列中的消息，但是特定消息只能由最多一个消费者消费
```
sender -> messageQueue -> receiver
```

## 发布订阅（pub-sub）  
消费者可以订阅一个或多个主题并使用该主题中的所有消息
```
sender -> messageQueue -> receiver1
                       |-> receiver2
                       |-> receiver3
```

## 数据生产（Produce）
一条记录 四个参数 topic(必须) patition key value(必须)。  
1. 序列化
2. 放进对应发送队列  
根据topic patition确定队列。若没有patition，根据hash key计算，相同则去同一个patition，没有key，Round-Robin 选patition。  
producer跟topic下所有partition leader保持socket连接。patition leader的位置（host:port）保存在zk中。producer作为客户端注册watch监听leader的变更事件。  
异步发送，多条消息暂存客户端的缓存中，批量发送至broker。

## 数据消费（Consume）
每个消费者属于一个consumer group，一个group包含多个consumer。订阅Topic是按照消费组来订阅，每个消息只会被组中的一个consumer消费。点对点系统实际就是只包含一个consumer的group。广播系统实际就是每个consumer具备不同的group。同一个消费组中的两个消费者不会同时消费一个partition。所以同个组中的消费者数量最好是小于等于partition。  
consumer主动pull(fetch) broker中的消息，consumer主动根据自己的能力去fetch并处理数据，可以控制消费的进度（offset）。  
接收消息后需要保存partition的Offset消费到哪里，0.10版本前保存在zk中，zk写慢，每隔一分钟上报一次，0.10版本之后，保存在consumeroffsets topic中。


## 消息传递机制
三种机制：最多一次（At most once）、至少一次（At least once）、只且一次（Exactly once）。常用At least once模型。
